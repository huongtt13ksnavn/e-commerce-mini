## Intended Test Type: Integration / API Tests

The dependency graph reveals the intended strategy:

- `Microsoft.AspNetCore.Mvc.Testing` → spin up the full ASP.NET Core pipeline in-process using `WebApplicationFactory<Program>` (note: `Program` is declared `public partial class Program;` in `src/ECommerce.API/Program.cs` for test visibility)
- `Testcontainers.PostgreSql` → spin up a real PostgreSQL container per test collection, replacing the production connection string
- `FluentAssertions` → assert on HTTP responses, JSON bodies, and status codes

**Expected test pattern (to be implemented):**

```csharp
public class ProductEndpointsTests : IAsyncLifetime
{
    private readonly PostgreSqlContainer _db = new PostgreSqlBuilder().Build();
    private HttpClient _client = null!;

    public async Task InitializeAsync()
    {
        await _db.StartAsync();
        var factory = new WebApplicationFactory<Program>()
            .WithWebHostBuilder(host =>
                host.UseSetting("ConnectionStrings:DefaultConnection", _db.GetConnectionString()));
        _client = factory.CreateClient();
    }

    public Task DisposeAsync() => _db.DisposeAsync().AsTask();

    [Fact]
    public async Task GetProducts_ReturnsOk()
    {
        var response = await _client.GetAsync("/api/products");
        response.StatusCode.Should().Be(HttpStatusCode.OK);
    }
}
```

## CI Integration

Tests run automatically via GitHub Actions on every push/PR to `master`/`main`.

**Workflow:** `.github/workflows/ci.yml`

```yaml
- name: Test
  run: dotnet test --no-build -c Release --logger "trx;LogFileName=test-results.xml" --results-directory ./TestResults
  env:
    DOCKER_HOST: unix:///var/run/docker.sock
```

`DOCKER_HOST` is set — the CI runner supports Docker, which is required by Testcontainers for PostgreSQL containers. Test results are uploaded as a build artifact (`TestResults/`) on every run (including failures).

## What to Test

Given the architecture, integration tests should cover:

- **API endpoint behavior** — correct HTTP status codes, response shapes, and error bodies for all routes in `src/ECommerce.API/Endpoints/`
- **Authentication/authorization** — `AdminOnly` policy enforcement on product write endpoints, 401/403 responses
- **Validation** — 422 responses for invalid request bodies (FluentValidation pipeline)
- **Domain error mapping** — 404 for not-found, 409 for concurrency conflicts, 400 for domain rule violations
- **Rate limiting** — 429 response on `/api/auth/register` and `/api/auth/login` after 5 requests/minute

**No unit test project exists.** Domain logic (e.g., `Product`, `Money`, `AggregateRoot`) and application behaviors are not currently covered by isolated unit tests.
