## Architecture Overview

```
┌─────────────────────────────────────────────────┐
│                   API Layer                     │
│   (Minimal APIs, Endpoints, Middleware)         │
├─────────────────────────────────────────────────┤
│              Application Layer                  │
│   (Commands, Queries, Handlers, Behaviors)      │
│   MediatR pipeline: ExceptionHandling →         │
│   Logging → Validation → Handler                │
├─────────────────────────────────────────────────┤
│               Domain Layer                      │
│   (Aggregates, Value Objects, Domain Events,    │
│    Repository Interfaces)                       │
├─────────────────────────────────────────────────┤
│            Infrastructure Layer                 │
│   (EF Core, Repository Impls, Event Dispatch,  │
│    JWT, PostgreSQL)                             │
└─────────────────────────────────────────────────┘
```

## Domain Model

### Aggregates and Value Objects

| Aggregate | Value Objects | Key Behaviors |
|-----------|--------------|---------------|
| `Product` | `ProductId`, `Money` (amount + currency), `ProductName` | Create, UpdatePrice, Deactivate |
| `Cart` | `CartId`, `CartItem` (ProductId + Quantity + UnitPrice) | AddItem, RemoveItem, Clear |
| `Order` | `OrderId`, `OrderItem` (ProductId + Quantity + UnitPrice), `Money` (total), `OrderStatus` | PlaceOrder, Cancel |

> Note: `User` is NOT a domain aggregate. Authentication and identity are handled by ASP.NET Identity. `UserId` is a value object (`record UserId(Guid Value)`) wrapping the `IdentityUser.Id` string parsed to `Guid`. It is used by `Cart` and `Order` to associate ownership without taking a dependency on Identity internals.

### Domain Events

| Event | Raised By | Handler(s) |
|-------|-----------|------------|
| `OrderPlaced` | `Order.PlaceOrder()` | `OrderPlacedNotificationHandler` — logs the event and publishes an `INotification` via MediatR (stub; no email/inventory in scope) |

> Note: `CartCheckedOut` is out of scope for MVP. The `PlaceOrderCommand` handler reads the Cart directly via `ICartRepository`, creates the Order, then calls `Cart.Clear()` in the same unit of work. No intermediate cart event is needed.

## API Contract

### Authentication
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/auth/register` | None | Register user |
| POST | `/api/auth/login` | None | Returns `{ token, expiresAt }` |
| GET | `/api/auth/me` | User JWT | Returns `{ id, email, role }` from JWT claims (no DB call) |

### Products (Catalog)
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/products` | None | Paginated list. Query params: `page`, `pageSize`, `search`. Response: `{ items: ProductDto[], total, page, pageSize }` |
| GET | `/api/products/{id}` | None | Get single product. Response: `ProductDto` (id, name, price, currency, isActive) |
| POST | `/api/products` | Admin JWT | Create product. Body: `{ name, price, currency }`. Response: `{ id }` |
| PUT | `/api/products/{id}` | Admin JWT | Update product price/name. Body: `{ name?, price? }`. Response: 204 |
| PATCH | `/api/products/{id}/deactivate` | Admin JWT | Deactivate product (calls `Product.Deactivate()`). Response: 204 |

### Cart
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/cart` | User JWT | Get current user's cart |
| POST | `/api/cart/items` | User JWT | Add item: `{ productId, quantity }` |
| DELETE | `/api/cart/items/{productId}` | User JWT | Remove item |
| DELETE | `/api/cart` | User JWT | Clear cart |

> **Cart ownership note:** Cart endpoints always operate on the requesting user's cart via `ICartRepository.GetByUserIdAsync(userId)` — `userId` is read from JWT claims. There is no `CartId` in any URL. Ownership is enforced by construction: no endpoint accepts a `CartId` parameter from the caller.

### Orders
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/orders` | User JWT | Place order from cart. Returns `{ orderId }`. Cart is cleared. Fires `OrderPlaced`. Edge cases: empty cart → 400 `"Cart is empty"`; inactive product in cart → 400 `"Product {id} is no longer available"` |
| GET | `/api/orders` | User JWT | List orders for current user. Response: `{ items: OrderSummaryDto[], total }`. `OrderSummaryDto`: `{ orderId, status, total, placedAt, items: [{ productName, quantity, unitPrice }] }` — fetched via EF Core `.Include(o => o.Items)` projection. One query, no N+1. |
| GET | `/api/orders/{id}` | User JWT | Get order detail. Response: `OrderDetailDto` (id, items, total, status, placedAt, cancelledAt). `IOrderRepository.GetByIdAsync(orderId, userId)` — returns null if not found or wrong user → 404. |
| PATCH | `/api/orders/{id}/cancel` | User JWT | Cancel order. Returns 204. 400 if order already completed. |

## Auth Specification

- **Token type:** JWT Bearer
- **Claims:** `sub` (UserId), `email`, `role` (User / Admin)
- **Token lifetime:** 1 hour
- **Refresh tokens:** Out of scope (MVP)
- **Protected routes:** All `/api/cart/*` and `/api/orders/*` require `User` or `Admin` role. `POST/PUT /api/products` requires `Admin` role.
- **Seed data:** One admin user seeded on startup (`admin@example.com / Admin123!`) and 10 sample products so the end-to-end demo flow (browse → cart → order) works without manual data entry

## MediatR Pipeline Behaviors (in execution order, outermost first)

1. `ExceptionHandlingBehavior<TRequest, TResponse>` — outermost wrapper; catches all unhandled exceptions (including `ValidationException` and domain exceptions) and maps to `ProblemDetails` responses
2. `LoggingBehavior<TRequest, TResponse>` — logs command/query name and elapsed time
3. `ValidationBehavior<TRequest, TResponse>` — runs all registered FluentValidation validators for `TRequest`; throws `ValidationException` with field errors on failure

> Ordering rationale: ExceptionHandling must be outermost so it catches exceptions from all inner behaviors. Logging wraps the full execution including validation time. Validation runs closest to the handler so it sees final request state.

## Database

- **Engine:** PostgreSQL 16 (via Docker)
- **ORM:** EF Core 10 with code-first migrations
- **Migration strategy:** `dbContext.Database.MigrateAsync()` called on application startup using Polly `WaitAndRetryAsync(5, attempt => TimeSpan.FromSeconds(attempt * 2))` — max 5 attempts with exponential backoff, then throws so the container exits with a clear error. Docker Compose uses `depends_on: db: condition: service_healthy` with a PostgreSQL healthcheck (`pg_isready -U postgres`) to ensure the database is ready before the app container starts. No manual migration steps.
- **Repository scope:** One repository interface per aggregate root (`IProductRepository`, `ICartRepository`, `IOrderRepository`) defined in the Domain layer. Implementations are in Infrastructure. Unit of Work (`IUnitOfWork` with `CommitAsync()`) wraps `DbContext.SaveChangesAsync()`. **The command handler calls `IUnitOfWork.CommitAsync()` as its final step** — not the repository, not a behavior. Queries (read side) call repositories directly with no Unit of Work.
- **Domain event dispatch:** `AppDbContext.SaveChangesAsync()` collects domain events from all tracked aggregate roots (via EF Core `ChangeTracker`), calls `base.SaveChangesAsync()`, then dispatches each collected event via MediatR `IPublisher.Publish()`. This ensures events fire only after the DB write commits. `IUnitOfWork.CommitAsync()` simply calls `dbContext.SaveChangesAsync()` — event dispatch is baked into the DbContext. Aggregates expose `IReadOnlyCollection<IDomainEvent> DomainEvents` and a `ClearDomainEvents()` method.

**Happy path (8 tests):**

2. `POST /api/auth/login` → returns JWT token
3. `GET /api/products` → returns paginated product list
4. `POST /api/products` (admin auth) → creates product, returns 201

**Critical edge cases (5 tests):**
9. `POST /api/orders` with empty cart → 400 `{ title: "Cart is empty" }`
10. `POST /api/orders` with inactive product in cart → 400 `{ title: "Product {id} is no longer available" }`
11. `GET /api/orders/{id}` using a different user's JWT → 404 (not 403)

## Implementation Pitfalls (Eng Review Notes)

These are non-obvious bugs that are easy to introduce. Each has an explicit fix.

1. **`Products` index.** Add `ix_products_active_name` covering `(IsActive, Name)` in EF Core `ProductConfiguration.cs`: `builder.HasIndex(p => new { p.IsActive, p.Name }).HasDatabaseName("ix_products_active_name")`. Cuts the paginated search scan to active products only.
2. **`OrderSummaryDto` — include product names via projection.** Use EF Core `.Include(o => o.Items)` to fetch order items in one query. `OrderSummaryDto` fields: `orderId`, `status`, `total`, `placedAt`, `items: [{ productName, quantity, unitPrice }]`. One query, no N+1.
3. **Cart item price is a snapshot.** `CartItem.UnitPrice` is set at add-time and never updated. `GET /api/cart` reads stored price — no product join needed. If the product price changes after the item is added, the cart shows the original price. This is correct e-commerce behavior. Do not "fix" this.
4. **Redis cache resilience.** Wrap all `IDistributedCache` reads in try/catch: `try { var cached = await _cache.GetStringAsync(key); } catch { /* fall through to DB */ }`. If Redis is down, the cache miss is silent and the DB serves the request. Never let a Redis outage cause a 500.
5. **Redis cache scope: single products only.** Cache key `product:{id}` for `GET /api/products/{id}`. Do not cache paginated list queries — too many key combinations. Cache invalidation: `RemoveAsync($"product:{id}")` in `UpdateProductCommandHandler` and `DeactivateProductCommandHandler` after commit.
6. **`DomainException` base class (Domain layer).** Define `abstract class DomainException : Exception` in Domain. Subclasses: `CartEmptyException`, `ProductUnavailableException(ProductId id)`, `CartNotFoundException(UserId userId)`, `OrderNotFoundException`. `ExceptionHandlingBehavior` catches `DomainException` → 400 ProblemDetails; catches all others → 500.
7. **`IOrderRepository.GetByIdAsync(OrderId, UserId)`** — always filters by both fields. Returns `null` if no match (order not found OR wrong user). Handler throws `OrderNotFoundException` → 404. Never 403 — do not leak that the order ID exists.
8. **EF Core entity configuration.** Domain entities must have zero EF annotations (`[Key]`, `[Required]`, `[Column]`). All mapping lives in `Infrastructure/Persistence/Configurations/{Entity}Configuration.cs` implementing `IEntityTypeConfiguration<T>`. Value objects configured as owned entities. Typed IDs (`ProductId`, `OrderId`, etc.) configured with value converters.
9. **`AuditInterceptor` — `CreatedAt` overwrite on update.** The interceptor must guard: `if (entity.CreatedAt == default) entity.CreatedAt = now;` — only set `CreatedAt` for new entities. Always set `UpdatedAt = now`. Without the guard, every `SaveChanges` resets `CreatedAt`.
10. **JWT secret — fail fast at startup.** Add `builder.Configuration["Jwt:Secret"] ?? throw new InvalidOperationException("Jwt:Secret is required")` in `Program.cs` before `AddAuthentication`. Without this, missing config causes silent auth failure on every request.
11. **Rate limiting middleware order.** `app.UseRateLimiter()` must be placed before `app.UseAuthentication()` in `Program.cs`. If reversed, unauthenticated requests bypass the rate limiter.
12. **`AddOpenApi()` required for Scalar.** Scalar reads the OpenAPI spec generated by `builder.Services.AddOpenApi()` (ASP.NET Core 9+ built-in). Must be called in `Program.cs` or Scalar renders an empty page.
